# 测试执行集成方案

更新时间：2026-03-17

## 文档范围

本文档说明执行域的设计目标、当前实现边界和后续演进方向。本文档不再使用“尚未落地”的表述替代当前实现。

## 1. 目标

- 从平台向执行代理下发任务
- 支持 HTTP 直连代理与可扩展的分发通道
- 接收任务事件、用例状态、任务完成结果
- 保证任务下发可追踪、可幂等、可审计

## 2. 当前实现边界

- `test_cases`：定义测试用例实体与业务编号
- `execution`：管理代理注册、任务下发、调度状态、消费确认、执行回传
- `workflow`：继续管理业务事项流转，不承载执行域的细粒度运行状态

当前实现目录以代码为准，核心包括：

```text
backend/app/modules/execution/
  api/
  schemas/
  application/
  service/
  repository/
```

## 3. 当前核心对象

### ExecutionTask

用于记录任务主状态，当前重点字段包括：

- `task_id`
- `external_task_id`
- `framework`
- `agent_id`
- `dispatch_channel`
- `dedup_key`
- `schedule_type`
- `schedule_status`
- `dispatch_status`
- `consume_status`
- `overall_status`
- `case_count`
- `planned_at`
- `triggered_at`
- `created_at`

### ExecutionTaskCase

用于记录任务内测试用例执行状态，当前重点字段包括：

- `task_id`
- `case_id`
- `status`
- `progress_percent`
- `step_total`
- `step_passed`
- `step_failed`
- `step_skipped`
- `last_seq`
- `started_at`
- `finished_at`

### ExecutionEvent

用于保存任务事件审计记录，当前重点字段包括：

- `task_id`
- `event_id`
- `event_type`
- `seq`
- `source_time`
- `received_at`
- `processed`

### ExecutionAgent

用于保存执行代理注册和心跳状态，当前重点字段包括：

- `agent_id`
- `hostname`
- `ip`
- `port`
- `base_url`
- `region`
- `status`
- `last_heartbeat_at`
- `heartbeat_ttl_seconds`
- `is_online`

## 4. 当前接口面

统一前缀：`/api/v1/execution`

### 代理接口

- `POST /agents/register`
- `POST /agents/{agent_id}/heartbeat`
- `GET /agents`
- `GET /agents/{agent_id}`

### 任务接口

- `POST /tasks/dispatch`
- `POST /tasks/{task_id}/consume-ack`
- `GET /tasks`
- `GET /tasks/{task_id}/status`
- `POST /tasks/{task_id}/cancel`
- `PUT /tasks/{task_id}/schedule`
- `POST /tasks/{task_id}/retry`

### 执行回传接口

- `POST /tasks/{task_id}/events`
- `POST /tasks/{task_id}/cases/{case_id}/status`
- `POST /tasks/{task_id}/complete`

## 5. 当前请求模型要点

### 任务下发 `POST /tasks/dispatch`

当前请求体以代码 schema 为准，核心字段包括：

- `framework`
- `agent_id`
- `trigger_source`
- `schedule_type`
- `planned_at`
- `callback_url`
- `dut`
- `cases`
- `runtime_config`

其中：

- `cases` 当前为 `{ "case_id": "..." }` 列表
- `agent_id` 在 HTTP 直连模式下是关键字段
- `schedule_type` 当前支持 `IMMEDIATE` 和 `SCHEDULED`

## 6. 当前约束

- 系统会生成 `task_id` 与 `external_task_id`
- 系统会生成 `dedup_key`，在消费者未确认前阻止相同业务命令重复下发
- 任务支持 `consume-ack` 机制，用于确认消费者已领取任务
- 代理在线状态由 `last_heartbeat_at + heartbeat_ttl_seconds` 推导
- 回传接口已按任务事件、用例状态、任务完成三个入口拆分
