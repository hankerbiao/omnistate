## 枚举值详解

### TestCaseCategory (测试类别)
- **FUNCTIONAL**: 功能测试 - 验证基本功能是否正常
- **STRESS**: 压力测试 - 验证系统在极限条件下的表现
- **PERFORMANCE**: 性能测试 - 验证系统性能指标
- **COMPATIBILITY**: 兼容性测试 - 验证软硬件兼容性
- **STABILITY**: 稳定性测试 - 验证长期运行稳定性
- **SECURITY**: 安全测试 - 验证安全性

### TestCaseStatus (用例状态)
- **DRAFT**: 草稿 - 刚创建，未提交评审
- **REVIEW**: 评审中 - 已提交，等待评审
- **APPROVED**: 已通过 - 评审通过，可以使用
- **DEPRECATED**: 已废弃 - 不再使用

### Priority (优先级)
- **P0**: 最高优先级 - 关键路径测试
- **P1**: 高优先级 - 重要功能测试
- **P2**: 普通优先级 - 常规测试

### RiskLevel (风险等级)
- **LOW**: 低风险 - 对系统无影响
- **MEDIUM**: 中等风险 - 可能影响部分功能
- **HIGH**: 高风险 - 可能导致系统崩溃

### VisibilityScope (可见范围)
- **TEAM**: 团队内 - 仅团队成员可见
- **PROJECT**: 项目内 - 项目内所有成员可见
- **GLOBAL**: 全局 - 所有用户可见

### Confidentiality (保密级别)
- **PUBLIC**: 公开 - 无保密要求
- **INTERNAL**: 内部 - 公司内部可见
- **NDA**: 保密协议 - 需要签署保密协议

## TestStep 结构

```typescript
interface TestStep {
  step_id: string;    // 步骤ID (例如: step-001)
  name: string;       // 步骤名称 (例如: 系统初始化)
  action: string;     // 执行操作 (例如: 启动系统并检查内存)
  expected: string;   // 预期结果 (例如: 系统识别所有内存模块)
}
```

## ApprovalRecord 结构

```typescript
interface ApprovalRecord {
  approver: string;     // 审批人ID
  timestamp: string;    // 审批时间 (ISO 8601)
  result: 'approved' | 'rejected' | 'commented';
  comment: string;      // 审批意见
}
```

## 环境要求结构

```typescript
interface RequiredEnv {
  os?: string;           // 操作系统 (例如: "Ubuntu 20.04")
  firmware?: string;     // 固件版本 (例如: "BIOS v2.1.0")
  hardware?: string;     // 硬件配置 (例如: "Intel Xeon Gold 6348")
  dependencies?: string[];  // 依赖组件 (例如: ["DDR5-5600"])
  tooling?: string[];    // 工具列表 (例如: ["MemTest86"])
}
```

## 表单验证要点

### 1. 字符长度
- title: 1-100字符
- change_log: 1-1000字符
- pre_condition: 1-2000字符
- post_condition: 1-2000字符
- failure_analysis: ≤1000字符
- deprecation_reason: ≤1000字符

### 2. 格式要求
- case_id: 建议 `TC-YYYY-NNN`
- time: ISO 8601 格式
- estimated_duration_sec: 正整数

### 3. 业务规则
- target_components: 不允许重复
- steps: 至少1个步骤
- cleanup_steps: 至少1个清理步骤
- steps中的step_id: 必须唯一
- ref_req_id: 必须引用存在的需求
- owner_id/reviewer_id: 必须真实存在

### 4. 关联规则
- 自动化负责人(auto_dev_id)通常是AUTOMATION角色
- 评审人(reviewer_id)通常是TPM角色
- 所有者(owner_id)通常是TESTER角色
