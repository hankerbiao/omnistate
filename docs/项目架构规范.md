# 项目架构规范（FastAPI 后端）

## 1. 目标与原则
- 面向大型项目可扩展：模块化、清晰边界、可并行开发。
- 领域驱动拆分：业务模块彼此隔离，跨模块交互通过明确契约完成。
- 分层清晰：`API`/`Service`/`Repository`/`Domain` 各司其职。
- 高可测试性：业务逻辑与框架、数据库解耦。
- 统一接口规范：响应格式、错误处理、日志与异常约束一致。

## 2. 目录结构规范
```
backend/app/
  main.py                       # 应用入口
  modules/                       # 业务模块聚合层
    workflow/                    # 工作流域（当前已落地）
      api/                       # HTTP 层
      schemas/                   # Pydantic 模型
      service/                   # 业务服务层
      domain/                    # 领域模型/异常/规则
      repository/                # 持久化封装
        models/                  # Beanie 文档模型
      docs/                      # 模块设计文档
    requirement/                 # 测试需求管理（预留）
    testcase/                    # 测试用例管理（预留）
    plan/                        # 测试计划管理（预留）
    execution/                   # 测试执行管理（预留）
    report/                      # 测试报告管理（预留）

  shared/                        # 共享基础设施层
    api/                         # 通用 API 组件
      main.py                    # 路由汇总入口
      errors/                    # 全局异常处理
      schemas/                   # 通用响应模型
      routes/                    # 共享路由（如 health）
    core/                        # 日志、客户端、通用工具
    db/                          # 数据库配置、连接器

  configs/                       # 配置文件（JSON/等）
  init_mongodb.py                # 初始化脚本
```

### 目录职责说明
- `modules/*`：业务边界清晰，避免跨模块直接访问数据库或模型。
- `shared/*`：框架/基础设施/跨模块通用能力。
- `main.py`：只负责组装（配置、中间件、路由、生命周期）。

## 3. 分层依赖规则（强约束）
**依赖方向：API → Service → Repository/Domain**

禁止行为：
- `api` 层直接访问数据库或 Beanie 文档。
- `service` 层引用 `fastapi` 请求对象或 HTTP 细节。
- 模块之间直接调用对方的 `repository` 或 `models`。

允许行为：
- `api` 调用 `service`。
- `service` 调用 `repository` 或领域规则。
- `repository` 只处理数据访问，不包含业务规则。

## 4. API 层规范
- 路由只处理 **请求解析、权限校验、参数校验、异常映射**。
- 所有路由返回统一 Envelope 格式（详见第 6 节）。
- API 路由必须使用 `response_model` 声明响应结构。

示例：
```python
@router.get("/items", response_model=APIResponse[List[ItemResponse]])
async def list_items(service: ItemServiceDep):
    data = await service.list_items()
    return APIResponse(data=data)
```

## 5. Service 层规范
- 仅包含业务逻辑，不感知 HTTP 细节。
- 所有业务异常抛出领域异常（`domain/exceptions.py`）。
- 允许聚合多仓储调用，但禁止出现 SQL/ODM 细节。

## 6. 统一响应 Envelope 规范
统一格式：
```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

规范说明：
- `code`：成功为 `0`，失败为 HTTP 状态码或业务码。
- `message`：成功默认 `ok`，失败为错误语义说明。
- `data`：成功为业务数据，失败为 `ErrorResponse`。

错误响应结构：
```json
{
  "error": "ErrorName",
  "detail": "human readable message"
}
```

## 7. 领域异常规范
- 所有业务异常集中在 `modules/<module>/domain/exceptions.py`。
- API 层只映射异常到 HTTP 状态码，异常含义由 `message` 输出。

## 8. Repository 层规范
- Repository 层负责数据库访问、查询构建与持久化。
- 禁止在 Repository 中处理跨实体复杂逻辑，复杂规则留在 Service。
- 强制输出“业务可用数据结构”，禁止直接返回 ORM/ODM 对象。

## 9. Schema 规范
- `Request` 与 `Response` 模型分离。
- 模型命名规则：`CreateXxxRequest`、`XxxResponse`。
- 任何跨模块复用的结构放入 `shared/api/schemas`。

## 10. 模块扩展规范
新增模块（例如 `requirement`、`testcase`）需按以下步骤：
1. 在 `modules/` 下创建对应目录。
2. 建立 `api/service/domain/repository/schemas` 子目录。
3. 在 `shared/api/main.py` 中注册路由。
4. 更新测试用例与文档。

## 11. 日志与监控
- 全局日志统一使用 `shared/core/logger.py`。
- 重要业务动作必须记录日志（创建/流转/改派/删除）。
- 所有异常必须记录堆栈日志，禁止吞异常。

## 12. 测试规范
- 单元测试：针对 `service` 与 `domain`。
- 集成测试：针对 `api`，验证响应结构与状态码。
- 建议按模块目录组织：`tests/modules/<module>/...`。

## 13. 版本与兼容
- API 统一前缀：`/api/v1`。
- 破坏性变更必须升级版本号。
- 保持对旧前端兼容时，新增字段不删除字段。

## 14. 配置与安全
- 所有环境配置统一来自 `shared/db/config.py` 与 `.env`。
- 禁止硬编码敏感配置。
- CORS 必须使用白名单，禁止生产使用 `*`。

## 15. 文档要求
- 每个模块维护 `docs/`。
- 接口接入说明统一集中在 `docs/接口与认证说明.md`，模块 API 细节维护在 `docs/api/`。
- 架构规范统一集中在 `docs/项目架构规范.md`。
