# 测试用例字段定义文档

## 概述

本文档详细说明了 Server Test Case Designer 系统中测试用例（TestCase）的所有字段定义、数据类型、约束条件和示例。

---

## 数据模型

### TestCase 接口

```typescript
export interface TestCase {
  case_id: string;
  ref_req_id: string;
  title: string;
  test_category: TestCaseCategory;
  version: number;
  is_active: boolean;
  change_log: string;
  status: TestCaseStatus;
  owner_id: string;
  reviewer_id: string;
  auto_dev_id?: string;
  priority: Priority;
  estimated_duration_sec: number;
  target_components: string[];
  required_env: {
    os?: string;
    firmware?: string;
    hardware?: string;
    dependencies?: string[];
    tooling?: string[];
  };
  tags: string[];
  pre_condition: string;
  post_condition: string;
  cleanup_steps: TestStep[];
  steps: TestStep[];
  is_need_auto: boolean;
  is_destructive: boolean;
  automation_type: string;
  script_entity_id: string;
  risk_level: RiskLevel;
  visibility_scope: VisibilityScope;
  confidentiality: Confidentiality;
  attachments: Attachment[];
  custom_fields: Record<string, string>;
  failure_analysis?: string;
  deprecation_reason?: string;
  approval_history: ApprovalRecord[];
  created_at: string;
  updated_at: string;
}
```

### TestStep 接口

```typescript
export interface TestStep {
  step_id: string;
  name: string;
  action: string;
  expected: string;
}
```

### ApprovalRecord 接口

```typescript
export interface ApprovalRecord {
  approver: string;
  timestamp: string;
  result: 'approved' | 'rejected' | 'commented';
  comment: string;
}
```

## 字段详细说明

### 1. 基本信息字段

#### 1.1 case_id (用例ID)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试用例的唯一标识符
- **格式**: 建议格式：`TC-YYYY-NNN` (例如：TC-2026-001)
- **示例**: `"TC-2026-001"`

#### 1.2 ref_req_id (关联需求ID)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 关联的测试需求ID
- **格式**: 必须匹配现有需求的 `req_id`
- **示例**: `"TR-2026-001"`

#### 1.3 title (用例标题)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试用例标题，应简洁明确地描述测试内容
- **最大长度**: 建议不超过100个字符
- **示例**: `"DDR5内存高温压力测试"`

#### 1.4 test_category (测试类别)
- **类型**: `TestCaseCategory` (枚举)
- **是否必填**: ✅ 是
- **描述**: 测试用例的分类
- **枚举值**:
  - `FUNCTIONAL` - 功能测试
  - `STRESS` - 压力测试
  - `PERFORMANCE` - 性能测试
  - `COMPATIBILITY` - 兼容性测试
  - `STABILITY` - 稳定性测试
  - `SECURITY` - 安全测试
- **示例**: `"STRESS"`

#### 1.5 version (版本号)
- **类型**: `number`
- **是否必填**: ✅ 是
- **描述**: 测试用例版本号
- **默认**: `1`
- **示例**: `1`, `2`, `3`

#### 1.6 is_active (是否激活)
- **类型**: `boolean`
- **是否必填**: ✅ 是
- **描述**: 测试用例是否激活可用
- **默认**: `true`
- **示例**: `true`

---

### 2. 状态与变更

#### 2.7 change_log (变更日志)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试用例变更记录
- **示例**: `"初始版本创建"`

#### 2.8 status (状态)
- **类型**: `TestCaseStatus` (枚举)
- **是否必填**: ✅ 是
- **描述**: 当前用例状态
- **枚举值**:
  - `DRAFT` - 草稿
  - `REVIEW` - 评审中
  - `APPROVED` - 已通过
  - `DEPRECATED` - 已废弃
- **示例**: `"DRAFT"`

---

### 3. 责任人

#### 2.9 owner_id (所有者ID)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试用例负责人的UserID
- **示例**: `"tester001"`

#### 2.10 reviewer_id (评审人ID)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 评审人的UserID
- **示例**: `"tpm001"`

#### 2.11 auto_dev_id (自动化负责人ID)
- **类型**: `string` (可选)
- **是否必填**: ❌ 否
- **描述**: 自动化脚本开发负责人的UserID
- **示例**: `"auto001"`

---

### 4. 优先级与执行

#### 2.12 priority (优先级)
- **类型**: `Priority` (枚举)
- **是否必填**: ✅ 是
- **描述**: 测试用例优先级
- **枚举值**: `P0`, `P1`, `P2`
- **示例**: `"P0"`

#### 2.13 estimated_duration_sec (预估时长)
- **类型**: `number`
- **是否必填**: ✅ 是
- **描述**: 预估执行时长（秒）
- **示例**: `3600` (1小时)

---

### 5. 测试目标与环境

#### 2.14 target_components (目标组件)
- **类型**: `string[]`
- **是否必填**: ✅ 是
- **描述**: 需要测试的硬件组件列表
- **示例**: `["Memory", "CPU"]`

#### 2.15 required_env (测试环境要求)
- **类型**: `object`
- **是否必填**: ✅ 是
- **描述**: 测试所需的环境配置
- **结构**:
  ```typescript
  {
    os?: string;           // 操作系统版本
    firmware?: string;     // 固件版本
    hardware?: string;     // 硬件配置
    dependencies?: string[];  // 依赖组件
    tooling?: string[];    // 工具列表
  }
  ```
- **示例**:
  ```json
  {
    "os": "Ubuntu 20.04",
    "firmware": "BIOS v2.1.0",
    "hardware": "Intel Xeon Gold 6348",
    "dependencies": ["DDR5-5600", "NVMe SSD"],
    "tooling": ["MemTest86", "Stress-ng"]
  }
  ```

---

### 6. 标签与条件

#### 2.16 tags (标签)
- **类型**: `string[]`
- **是否必填**: ✅ 是
- **描述**: 测试用例标签，用于分类和搜索
- **示例**: `["内存", "高温", "压力测试"]`

#### 2.17 pre_condition (前置条件)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试执行前需要满足的条件
- **示例**: `"系统已安装DDR5内存，BIOS已更新至v2.1.0"`

#### 2.18 post_condition (后置条件)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试执行后需要恢复的状态
- **示例**: `"系统重启至正常配置，移除所有测试负载"`

---

### 7. 测试步骤

#### 2.19 cleanup_steps (清理步骤)
- **类型**: `TestStep[]`
- **是否必填**: ✅ 是
- **描述**: 测试后的清理操作步骤
- **结构**: 同 `steps` 字段
- **示例**:
  ```json
  [
    {
      "step_id": "cleanup-001",
      "name": "停止测试负载",
      "action": "结束所有压力测试进程",
      "expected": "CPU使用率降至5%以下"
    }
  ]
  ```

#### 2.20 steps (测试步骤)
- **类型**: `TestStep[]`
- **是否必填**: ✅ 是
- **描述**: 测试执行的具体步骤
- **结构**:
  ```typescript
  {
    step_id: string;    // 步骤ID
    name: string;       // 步骤名称
    action: string;     // 执行操作
    expected: string;   // 预期结果
  }
  ```
- **示例**:
  ```json
  [
    {
      "step_id": "step-001",
      "name": "系统初始化",
      "action": "启动系统并检查内存检测",
      "expected": "系统识别所有DDR5内存模块"
    },
    {
      "step_id": "step-002",
      "name": "运行压力测试",
      "action": "使用MemTest86进行4小时高温测试",
      "expected": "错误率为0，系统稳定运行"
    }
  ]
  ```

### 8. 自动化相关

#### 2.21 is_need_auto (是否需要自动化)
- **类型**: `boolean`
- **是否必填**: ✅ 是
- **描述**: 是否需要开发自动化脚本
- **示例**: `true`

#### 2.22 is_destructive (是否破坏性测试)
- **类型**: `boolean`
- **是否必填**: ✅ 是
- **描述**: 测试是否会对系统造成破坏性影响
- **示例**: `true`

#### 2.23 automation_type (自动化类型)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 自动化实现方式
- **示例**: `"pytest"`, `"robotframework"`, `"shell_script"`

#### 2.24 script_entity_id (脚本实体ID)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 关联的自动化脚本ID
- **示例**: `"script-001"`

### 9. 风险与可见性

#### 2.25 risk_level (风险等级)
- **类型**: `RiskLevel` (枚举)
- **是否必填**: ✅ 是
- **描述**: 测试用例的风险等级
- **枚举值**:
  - `LOW` - 低风险
  - `MEDIUM` - 中等风险
  - `HIGH` - 高风险
- **示例**: `"HIGH"`

#### 2.26 visibility_scope (可见范围)
- **类型**: `VisibilityScope` (枚举)
- **是否必填**: ✅ 是
- **描述**: 测试用例的可见范围
- **枚举值**:
  - `TEAM` - 团队内
  - `PROJECT` - 项目内
  - `GLOBAL` - 全局
- **示例**: `"PROJECT"`

#### 2.27 confidentiality (保密级别)
- **类型**: `Confidentiality` (枚举)
- **是否必填**: ✅ 是
- **描述**: 测试用例的保密级别
- **枚举值**:
  - `PUBLIC` - 公开
  - `INTERNAL` - 内部
  - `NDA` - 保密协议
- **示例**: `"INTERNAL"`

### 10. 附件与扩展

#### 2.28 attachments (附件)
- **类型**: `Attachment[]`
- **是否必填**: ✅ 是 (可以为空数组)
- **描述**: 相关附件列表
- **结构**: 同需求模块的Attachment
- **示例**:
  ```json
  [
    {
      "id": "att-001",
      "name": "test_script.py",
      "type": "other",
      "url": "https://example.com/test_script.py",
      "size": "5KB",
      "uploaded_at": "2026-01-15T08:00:00Z"
    }
  ]
  ```

#### 2.29 custom_fields (自定义字段)
- **类型**: `Record<string, string>`
- **是否必填**: ✅ 是
- **描述**: 自定义扩展字段
- **示例**:
  ```json
  {
    "test_environment": "lab_A",
    "operator": "张三",
    "test_batch": "batch_001"
  }
  ```

- **描述**: 测试失败后的分析记录
- **示例**: `"高温下ECC错误率超出预期"`

#### 2.31 deprecation_reason (废弃原因)
- **类型**: `string` (可选)
- **是否必填**: ❌ 否
- **描述**: 废弃用例的原因
- **示例**: `"已由新用例替代"`

- **类型**: `ApprovalRecord[]`
- **是否必填**: ✅ 是
- **描述**: 审批记录历史


#### 2.33 created_at (创建时间)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 用例创建时间
- **示例**: `"2026-01-15T08:00:00Z"`

#### 2.34 updated_at (更新时间)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 用例最后更新时间
- **示例**: `"2026-02-20T10:30:00Z"`

## 工作流状态说明

### 用例生命周期

```
DRAFT (草稿)
    ↓ 提交评审
REVIEW (评审中)
    ↓ 评审通过
APPROVED (已通过)
    ↓ 废弃
DEPRECATED (已废弃)
```

### 状态转换规则

| 当前状态 | 允许操作 | 下一状态 | 操作人 |
|----------|----------|----------|--------|
| DRAFT | 提交评审 | REVIEW | 所有者 |
| REVIEW | 评审通过 | APPROVED | 评审人 |
| REVIEW | 评审不通过 | DRAFT | 评审人 |
| REVIEW | 评审意见 | REVIEW | 评审人 |
| APPROVED | 废弃 | DEPRECATED | 所有者/管理员 |
| DEPRECATED | - | 终止 | - |

## 表单验证规则

### 必填字段验证
- ✅ `case_id`: 不能为空，必须唯一
- ✅ `ref_req_id`: 必须引用存在的需求ID
- ✅ `title`: 不能为空，长度 1-100 字符
- ✅ `test_category`: 必须选择有效类别
- ✅ `version`: 必须为正整数
- ✅ `is_active`: 布尔值
- ✅ `change_log`: 不能为空
- ✅ `status`: 必须选择有效状态
- ✅ `owner_id`: 必须指定有效的用户ID
- ✅ `reviewer_id`: 必须指定有效的用户ID
- ✅ `priority`: 必须选择 P0/P1/P2 之一
- ✅ `estimated_duration_sec`: 必须为正整数
- ✅ `target_components`: 至少选择 1 个组件
- ✅ `required_env`: 必须包含有效配置
- ✅ `tags`: 至少包含 1 个标签
- ✅ `pre_condition`: 不能为空
- ✅ `post_condition`: 不能为空
- ✅ `cleanup_steps`: 至少包含 1 个清理步骤
- ✅ `steps`: 至少包含 1 个测试步骤
- ✅ `is_need_auto`: 布尔值
- ✅ `is_destructive`: 布尔值
- ✅ `automation_type`: 不能为空
- ✅ `script_entity_id`: 不能为空
- ✅ `risk_level`: 必须选择有效等级
- ✅ `visibility_scope`: 必须选择有效范围
- ✅ `confidentiality`: 必须选择有效级别
- ✅ `attachments`: 数组
- ✅ `custom_fields`: 对象
- ✅ `approval_history`: 数组

### 业务规则验证
1. `ref_req_id` 必须引用存在的需求
2. `owner_id` 和 `reviewer_id` 可以是同一用户
3. `target_components` 不允许包含重复值
4. `steps` 和 `cleanup_steps` 中的 `step_id` 必须唯一
5. `approval_history` 中的 `approver` 必须真实存在
6. `custom_fields` 的键值都不能为空

## API 端点

### 创建用例
```
POST /api/v1/test-cases
Content-Type: application/json

{
  // 用例对象（不含 case_id, version, created_at, updated_at）
}
```

### 获取用例列表
```
GET /api/v1/test-cases
Query Parameters:
  - page: 页码 (默认: 1)
  - page_size: 每页数量 (默认: 20)
  - status: 状态过滤
  - priority: 优先级过滤
  - test_category: 类别过滤
  - owner_id: 所有者过滤
  - ref_req_id: 关联需求过滤
```

### 获取用例详情
```
GET /api/v1/test-cases/{case_id}
```

### 更新用例
```
PUT /api/v1/test-cases/{case_id}
Content-Type: application/json

{
  // 用例对象
}
```

### 删除用例
```
DELETE /api/v1/test-cases/{case_id}
```

### 提交评审
```
POST /api/v1/test-cases/{case_id}/submit-review
```

### 审批用例
```
POST /api/v1/test-cases/{case_id}/approve
Content-Type: application/json

{
  "result": "approved" | "rejected" | "commented",
  "comment": "审批意见"
}
```
