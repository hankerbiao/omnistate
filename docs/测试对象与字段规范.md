# 测试对象与字段规范

更新时间：2026-03-17

## 文档范围

本文档是测试需求、测试用例及其创建字段的唯一入口，合并了原字段总览、对象字段定义和前后端字段统一说明。

适用范围：

- 产品和研发对齐对象边界
- 前端创建表单建模
- 后端接口字段校验

## 1. 对象关系

### 测试需求（Requirement）

- 回答“测什么”
- 业务主键：`req_id`
- 可关联多个测试用例
- 可按目标组件/BOM 做覆盖分析

### 测试用例（Test Case）

- 回答“怎么测”
- 业务主键：`case_id`
- 通过 `ref_req_id` 关联测试需求
- 维护步骤、评审状态和自动化关联信息

## 2. 最小创建字段

### 创建测试需求 `POST /api/v1/requirements`

建议最小字段：

- `req_id`
- `title`
- `description`
- `target_components`
- `priority`
- `key_parameters`
- `tpm_owner_id`

常用可选字段：

- `technical_spec`
- `firmware_version`
- `risk_points`
- `manual_dev_id`
- `auto_dev_id`
- `attachments`

创建时禁止提交：

- `status`
- `created_at`
- `updated_at`
- `workflow_item_id`

### 创建测试用例 `POST /api/v1/test-cases`

建议最小字段：

- `case_id`
- `ref_req_id`
- `title`
- `test_category`
- `owner_id`
- `reviewer_id`
- `priority`
- `steps`

常用可选字段：

- `version`
- `estimated_duration_sec`
- `target_components`
- `required_env`
- `tooling_req`
- `tags`
- `pre_condition`
- `post_condition`
- `cleanup_steps`
- `is_need_auto`
- `is_automated`
- `automation_type`
- `script_entity_id`
- `attachments`

创建时禁止提交：

- `status`
- `created_at`
- `updated_at`
- `workflow_item_id`

## 3. 测试需求字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `req_id` | `string` | 是 | 需求业务编号，建议唯一 |
| `title` | `string` | 是 | 需求标题 |
| `description` | `string` | 是 | 需求描述 |
| `technical_spec` | `string \| null` | 否 | 技术规格 |
| `target_components` | `string[]` | 是 | 目标组件/BOM 范围 |
| `firmware_version` | `string \| null` | 否 | 相关固件版本 |
| `priority` | `P0 \| P1 \| P2` | 是 | 优先级 |
| `key_parameters` | `{key:string,value:string}[]` | 是 | 关键参数 |
| `risk_points` | `string \| null` | 否 | 风险说明 |
| `tpm_owner_id` | `string` | 是 | TPM 负责人 |
| `manual_dev_id` | `string \| null` | 否 | 手工用例开发人 |
| `auto_dev_id` | `string \| null` | 否 | 自动化开发人 |
| `status` | `PENDING \| DEVELOPING \| REVIEWING \| CLOSED` | 否 | 工作流状态，由后端维护 |
| `attachments` | `object[]` | 否 | 附件列表 |
| `created_at` | `datetime string` | 否 | 创建时间 |
| `updated_at` | `datetime string` | 否 | 更新时间 |

约束：

- `req_id` 必须稳定可检索，不能依赖数据库内部主键。
- `target_components` 是需求与 BOM 覆盖分析的关键字段。
- `status` 由工作流驱动，前端创建/编辑时不应自行写入终态。

## 4. 测试用例字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `case_id` | `string` | 是 | 用例业务编号 |
| `ref_req_id` | `string` | 是 | 关联需求编号 |
| `title` | `string` | 是 | 用例标题 |
| `test_category` | `string` | 是 | 类别，如功能/性能/稳定性 |
| `version` | `number` | 否 | 版本号，默认可为 `1` |
| `is_active` | `boolean` | 否 | 是否有效 |
| `change_log` | `string \| null` | 否 | 变更说明 |
| `status` | `DRAFT \| REVIEW \| APPROVED \| DEPRECATED` | 否 | 用例状态，由后端维护 |
| `owner_id` | `string \| null` | 否 | 负责人 |
| `reviewer_id` | `string \| null` | 否 | 评审人 |
| `auto_dev_id` | `string \| null` | 否 | 自动化负责人 |
| `priority` | `P0 \| P1 \| P2 \| null` | 否 | 优先级 |
| `estimated_duration_sec` | `number \| null` | 否 | 预估时长 |
| `target_components` | `string[]` | 否 | 目标组件 |
| `required_env` | `object` | 否 | 运行环境要求 |
| `tooling_req` | `string[]` | 否 | 所需工具，统一为顶层字段 |
| `tags` | `string[]` | 否 | 标签 |
| `pre_condition` | `string \| null` | 否 | 前置条件 |
| `post_condition` | `string \| null` | 否 | 后置条件 |
| `cleanup_steps` | `TestStep[]` | 否 | 清理步骤 |
| `steps` | `TestStep[]` | 是 | 执行步骤 |
| `is_need_auto` | `boolean` | 否 | 是否需要自动化 |
| `is_automated` | `boolean` | 否 | 是否已自动化 |
| `automation_type` | `string \| null` | 否 | 自动化类型 |
| `script_entity_id` | `string \| null` | 否 | 脚本实体标识 |
| `risk_level` | `string \| null` | 否 | 风险级别 |
| `failure_analysis` | `string \| null` | 否 | 失败分析 |
| `confidentiality` | `string \| null` | 否 | 保密级别 |
| `visibility_scope` | `string \| null` | 否 | 可见范围 |
| `attachments` | `object[]` | 否 | 附件 |
| `custom_fields` | `object` | 否 | 扩展字段 |
| `deprecation_reason` | `string \| null` | 否 | 废弃原因 |
| `approval_history` | `object[]` | 否 | 审批记录 |
| `created_at` | `datetime string` | 否 | 创建时间 |
| `updated_at` | `datetime string` | 否 | 更新时间 |

`TestStep` 建议结构：

```json
{
  "step_id": "step-001",
  "name": "运行步骤",
  "action": "执行具体操作",
  "expected": "预期结果"
}
```

约束：

- `ref_req_id` 必须指向已存在的需求。
- `tooling_req` 作为工具要求的统一字段，不再使用 `required_env.tooling`。
- 自动化关联优先走显式关联字段，`script_entity_id` 仅作为执行层脚本标识。

## 5. 前后端统一规则

- 登录请求统一使用 `user_id`，不使用 `username` 作为接口字段。
- 创建接口只提交 `Create*Request` 范围内字段，不夹带状态、时间戳、工作流字段。
- 状态值以工作流/后端枚举为准，前端只展示，不自行定义新状态。
- 前端表单应区分“必填最小集”和“扩展补充字段”，避免一次性暴露所有字段。
