# 测试需求字段定义文档

## 概述

本文档详细说明了 Server Test Case Designer 系统中测试需求（TestRequirement）的所有字段定义、数据类型、约束条件和示例。

---

## 数据模型

### TestRequirement 接口

```typescript
export interface TestRequirement {
  req_id: string;
  title: string;
  description: string;
  technical_spec?: string;
  target_components: string[];
  firmware_version?: string;
  priority: Priority;
  key_parameters: { key: string; value: string }[];
  risk_points?: string;
  tpm_owner_id: string;
  manual_dev_id: string;
  status: RequirementStatus;
  attachments: Attachment[];
  created_at: string;
  updated_at: string;
}
```

---

## 字段详细说明

### 1. 基本信息字段

#### 1.1 req_id (需求ID)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试需求的唯一标识符
- **格式**: 建议格式：`TR-YYYY-NNN` (例如：TR-2026-001)
- **示例**: `"TR-2026-001"`

#### 1.2 title (需求标题)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 需求标题，应简洁明确地描述测试内容
- **最大长度**: 建议不超过100个字符
- **示例**: `"DDR5 内存高温压力测试"`

#### 1.3 description (需求描述)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 详细描述测试需求、目的、测试范围等
- **示例**: `"针对新一代服务器平台的 DDR5 内存进行全面的压力与兼容性验证，确保在高温高负载下的稳定性。重点关注 ECC 错误率、内存带宽衰减和系统稳定性。"`

---

### 2. 技术规格字段

#### 2.4 technical_spec (技术规格)
- **类型**: `string` (可选)
- **是否必填**: ❌ 否
- **描述**: 详细的技术规格说明
- **示例**: `"符合 JEDEC DDR5 标准，支持 5600MT/s 速率，工作电压 1.1V"`

#### 2.5 target_components (目标组件)
- **类型**: `string[]`
- **是否必填**: ✅ 是
- **描述**: 需要测试的硬件组件列表
- **枚举值**: `Memory`, `CPU`, `Storage`, `Network`, `GPU`, `BMC`, `BIOS`, `Thermal Sensor`, `Power Supply`, `Motherboard`
- **示例**: `["Memory", "CPU", "Thermal Sensor"]`

#### 2.6 firmware_version (固件版本)
- **类型**: `string` (可选)
- **是否必填**: ❌ 否
- **描述**: 相关的固件版本信息
- **格式**: `{组件名} v{版本号}`
- **示例**: `"BIOS v2.1.0"`, `"BMC v3.2.1"`

---

### 3. 优先级与参数

#### 2.7 priority (优先级)
- **类型**: `Priority` (枚举)
- **是否必填**: ✅ 是
- **描述**: 需求优先级
- **枚举值**:
  - `P0` - 最高优先级，关键路径测试
  - `P1` - 高优先级，重要功能测试
  - `P2` - 普通优先级，常规测试
- **示例**: `"P0"`

#### 2.8 key_parameters (关键参数)
- **类型**: `{ key: string, value: string }[]`
- **是否必填**: ✅ 是
- **描述**: 测试的关键参数列表
- **示例**:
  ```json
  [
    { "key": "电压", "value": "1.1V" },
    { "key": "频率", "value": "5600MT/s" },
    { "key": "温度", "value": "85°C" }
  ]
  ```

---

### 4. 风险与管理

#### 2.9 risk_points (风险点)
- **类型**: `string` (可选)
- **是否必填**: ❌ 否
- **描述**: 需要特别关注的测试风险点
- **示例**: `"重点验证高温下的 ECC 错误率，需注意不同内存厂商芯片的兼容性差异"`

---

### 5. 责任人与状态

#### 2.10 tpm_owner_id (TPM负责人)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 测试项目经理的UserID
- **示例**: `"tpm001"`

#### 2.11 manual_dev_id (手工执行负责人)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 手工测试执行负责人的UserID
- **示例**: `"tester001"`

#### 2.12 status (状态)
- **类型**: `RequirementStatus` (枚举)
- **是否必填**: ✅ 是
- **描述**: 当前需求状态
- **枚举值**:
  - `PENDING` - 待指派
  - `DEVELOPING` - 用例开发中
  - `REVIEWING` - 评审中
  - `CLOSED` - 已闭环
- **示例**: `"DEVELOPING"`

---

### 6. 附件与时间戳

#### 2.13 attachments (附件)
- **类型**: `Attachment[]`
- **是否必填**: ✅ 是 (可以为空数组)
- **描述**: 相关附件列表
- **结构**:
  ```typescript
  interface Attachment {
    id: string;           // 附件唯一ID
    name: string;         // 文件名
    type: 'image' | 'video' | 'spec' | 'log' | 'other';  // 文件类型
    url: string;          // 文件URL
    size?: string;        // 文件大小
    uploaded_at: string;  // 上传时间
  }
  ```
- **示例**:
  ```json
  [
    {
      "id": "att-001",
      "name": "DDR5_Spec.pdf",
      "type": "spec",
      "url": "https://example.com/DDR5_Spec.pdf",
      "size": "2.5MB",
      "uploaded_at": "2026-01-15T08:00:00Z"
    }
  ]
  ```

#### 2.14 created_at (创建时间)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 需求创建时间（ISO 8601格式）
- **示例**: `"2026-01-15T08:00:00Z"`

#### 2.15 updated_at (更新时间)
- **类型**: `string`
- **是否必填**: ✅ 是
- **描述**: 需求最后更新时间（ISO 8601格式）
- **示例**: `"2026-02-20T10:30:00Z"`

## 工作流状态说明

### 需求生命周期

```
PENDING (待指派)
    ↓ 指派TPM和测试工程师
DEVELOPING (用例开发中)
    ↓ 用例编写完成，提交评审
REVIEWING (评审中)
    ↓ 评审通过，需求闭环
CLOSED (已闭环)
```

### 状态转换规则

| 当前状态 | 允许操作 | 下一状态 | 操作人 |
|----------|----------|----------|--------|
| PENDING | 指派负责人 | DEVELOPING | 管理员/TPM |
| DEVELOPING | 提交评审 | REVIEWING | 手工执行负责人 |
| DEVELOPING | 重新指派 | PENDING | TPM |
| REVIEWING | 评审通过 | CLOSED | 管理员/TPM |
| REVIEWING | 评审不通过 | DEVELOPING | 管理员/TPM |
| CLOSED | - | 终止 | - |

## 表单验证规则

### 必填字段验证
- ✅ `req_id`: 不能为空，必须唯一
- ✅ `title`: 不能为空，长度 1-100 字符
- ✅ `description`: 不能为空，长度 1-2000 字符
- ✅ `target_components`: 至少选择 1 个组件
- ✅ `priority`: 必须选择 P0/P1/P2 之一
- ✅ `key_parameters`: 至少包含 1 个参数对
- ✅ `tpm_owner_id`: 必须指定有效的用户ID
- ✅ `manual_dev_id`: 必须指定有效的用户ID
- ✅ `status`: 必须选择有效状态

### 可选字段验证
- ❌ `technical_spec`: 长度不超过 2000 字符
- ❌ `firmware_version`: 格式：`组件名 版本号`，例如 "BIOS v1.0.0"
- ❌ `risk_points`: 长度不超过 1000 字符
- ❌ `attachments`: 文件类型必须是允许的类型之一

### 业务规则验证
1. `req_id` 格式检查：建议使用 `TR-YYYY-NNN` 格式
2. `tpm_owner_id` 和 `manual_dev_id` 不能是同一个用户（需要角色分离）
3. `target_components` 不允许包含重复值
4. `key_parameters` 中的 `key` 和 `value` 都不能为空

## API 端点

### 创建需求
```
POST /api/v1/requirements
Content-Type: application/json

{
  // 需求对象（不含 req_id, created_at, updated_at）
}
```

### 获取需求列表
```
GET /api/v1/requirements
Query Parameters:
  - page: 页码 (默认: 1)
  - page_size: 每页数量 (默认: 20)
  - status: 状态过滤
  - priority: 优先级过滤
  - tpm_owner_id: TPM负责人过滤
```

### 获取需求详情
```
GET /api/v1/requirements/{req_id}
```

### 更新需求
```
PUT /api/v1/requirements/{req_id}
Content-Type: application/json

{
  // 需求对象
}
```

### 删除需求
```
DELETE /api/v1/requirements/{req_id}
```
