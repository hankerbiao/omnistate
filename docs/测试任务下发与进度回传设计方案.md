# 测试任务下发与进度回传设计方案

更新时间：2026-03-03

## 1. 背景与目标

当前项目负责测试用例管理（`test_specs`），测试执行由外部测试框架完成。

需要新增两类能力：

1. 任务下发结果管理：
- 从当前项目将任务下发给外部框架。
- 记录下发请求、下发结果、外部任务标识、失败原因等。

2. 接收测试框架进度回报：
- 外部框架将任务状态、用例状态、步骤执行结果回传到当前项目。
- 当前项目落库并可查询，支撑前端任务看板/详情页。

本方案目标：
- 在不破坏现有 `workflow` 与 `test_specs` 的前提下，新增“执行域（execution）”。
- 保证回调接口幂等、安全、可追踪。
- 支持后续扩展（重试、重放、多框架、报表）。

---

## 2. 范围与非目标

### 2.1 本期范围（MVP）

- 新增执行域模块：任务下发、任务查询、进度回调。
- 任务粒度跟踪：任务总体状态 + 每个测试用例执行状态 + 步骤结果摘要。
- 提供可供前端查询的 API。

### 2.2 非目标（本期不做）

- 不改造现有 `workflow` 状态机配置驱动逻辑。
- 不实现复杂调度器（如队列消费、批量并发调度编排）。
- 不实现完整报表引擎（仅保留可扩展字段）。
- 不实现自动化用例库独立生命周期（当前仍以 `test_cases` 为下发源）。

---

## 3. 与现有架构的关系

### 3.1 现状

- `test_specs` 负责测试用例定义，`TestCaseDoc` 有业务唯一标识 `case_id`。
- `workflow` 负责事项流转。

### 3.2 新增执行域定位

新增模块：`backend/app/modules/execution/`

建议目录：

```text
execution/
  api/
    __init__.py
    routes.py
  schemas/
    __init__.py
    execution.py
  service/
    __init__.py
    execution_service.py
  repository/
    __init__.py
    models/
      __init__.py
      execution.py
```

职责边界：
- `test_specs`：定义“要测什么”。
- `execution`：记录“怎么下发、执行到哪里、结果如何”。
- `workflow`：仍用于业务事项流转，不强绑定执行进度状态。

## 4. 核心业务对象设计

## 4.1 ExecutionTask（任务主表）

集合：`execution_tasks`

关键字段：
- `task_id: str`：平台内任务唯一 ID（如 `ET-2026-000001`）
- `external_task_id: Optional[str]`：外部框架任务 ID
- `framework: str`：框架标识（如 `pytest-runner`）
- `dispatch_status: str`：下发状态
: `PENDING | DISPATCHED | DISPATCH_FAILED`
- `overall_status: str`：执行总体状态
: `QUEUED | RUNNING | PASSED | FAILED | PARTIAL_FAILED | CANCELLED | TIMEOUT`
- `request_payload: Dict[str, Any]`：下发请求快照
- `dispatch_response: Dict[str, Any]`：框架返回快照（成功/失败）
- `dispatch_error: Optional[str]`
- `created_by: str`
- `case_count: int`
- `reported_case_count: int`
- `started_at: Optional[datetime]`
- `finished_at: Optional[datetime]`
- `last_callback_at: Optional[datetime]`
- `is_deleted: bool`
- `created_at, updated_at`

索引建议：
- `task_id` 唯一索引
- `external_task_id` 普通索引
- `(created_by, created_at desc)`
- `(overall_status, created_at desc)`
- `(dispatch_status, created_at desc)`

## 4.2 ExecutionTaskCase（任务-用例明细）

集合：`execution_task_cases`

关键字段：
- `task_id: str`
- `case_id: str`（引用 `test_cases.case_id`）
- `case_snapshot: Dict[str, Any]`（标题、版本、关键配置快照）
- `status: str`
: `QUEUED | RUNNING | PASSED | FAILED | SKIPPED | BLOCKED | ERROR`
- `progress_percent: Optional[float]`
- `step_total: int`
- `step_passed: int`
- `step_failed: int`
- `step_skipped: int`
- `last_seq: int`（用于乱序保护）
- `last_event_id: Optional[str]`
- `started_at, finished_at, updated_at`

索引建议：
- `(task_id, case_id)` 唯一索引
- `(task_id, status)`

## 4.3 ExecutionEvent（回调事件审计）

集合：`execution_events`

关键字段：
- `task_id: str`
- `event_id: str`（框架传入，全局唯一或任务内唯一）
- `event_type: str`
: `TASK_STATUS | CASE_STATUS | STEP_RESULT | HEARTBEAT | SUMMARY`
- `seq: int`（框架事件序号）
- `received_at: datetime`
- `source_time: Optional[datetime]`
- `raw_payload: Dict[str, Any]`
- `processed: bool`
- `process_error: Optional[str]`

索引建议：
- `(task_id, event_id)` 唯一索引（幂等核心）
- `(task_id, seq)`
- `received_at`（可选 TTL，如保留 180 天）

## 5. 状态机设计

## 5.1 下发状态（dispatch_status）

- `PENDING`：已创建任务，未完成下发
- `DISPATCHED`：下发成功，拿到外部任务 ID（可为空，视框架）
- `DISPATCH_FAILED`：下发失败（网络/鉴权/框架拒绝）

允许迁移：
- `PENDING -> DISPATCHED`
- `PENDING -> DISPATCH_FAILED`
- `DISPATCH_FAILED -> DISPATCHED`（重试成功时）

## 5.2 总体执行状态（overall_status）

- 初始：`QUEUED`
- 中间：`RUNNING`
- 终态：`PASSED | FAILED | PARTIAL_FAILED | CANCELLED | TIMEOUT`

推荐规则：
- 任一用例失败且存在通过：`PARTIAL_FAILED`
- 全通过：`PASSED`
- 全失败：`FAILED`
- 框架显式取消/超时按回调值直接落地

## 5.3 用例执行状态（ExecutionTaskCase.status）

- `QUEUED -> RUNNING -> {PASSED|FAILED|SKIPPED|BLOCKED|ERROR}`

乱序保护规则：
- 仅处理 `seq > last_seq` 的状态事件。
- 小于等于 `last_seq` 的事件记录为重复/过期，不覆盖当前状态。

## 6. API 设计

统一前缀：`/api/v1/execution`

## 6.1 任务下发接口

### POST `/api/v1/execution/tasks/dispatch`

用途：创建执行任务并发起下发，返回下发结果。

请求体（示例）：

```json
{
  "framework": "pytest-runner",
  "trigger_source": "manual",
  "callback_url": "https://xxx/api/v1/execution/callbacks/progress",
  "dut": {
    "asset_id": "DUT-001",
    "bmc_ip": "10.0.0.1"
  },
  "cases": [
    {"case_id": "TC-2026-001"},
    {"case_id": "TC-2026-002"}
  ],
  "runtime_config": {
    "timeout_sec": 3600,
    "env": "staging"
  }
}
```

响应体（示例）：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "task_id": "ET-2026-000001",
    "external_task_id": "FW-abc-123",
    "dispatch_status": "DISPATCHED",
    "overall_status": "QUEUED",
    "case_count": 2,
    "created_at": "2026-03-03T10:00:00Z"
  }
}
```

错误场景：
- 400：`cases` 为空、存在非法字段、参数不合法
- 404：包含不存在的 `case_id`
- 409：同幂等键重复请求冲突（可选）
- 502：下发外部框架失败（但平台任务已落库为 `DISPATCH_FAILED`）

## 6.2 任务查询接口

### GET `/api/v1/execution/tasks/{task_id}`

返回任务主信息 + 聚合统计（通过/失败/运行中数量）。

### GET `/api/v1/execution/tasks/{task_id}/cases`

返回任务下用例执行明细，支持分页/状态过滤。

### GET `/api/v1/execution/tasks`

按 `created_by/framework/overall_status/dispatch_status/time range` 查询任务列表。

## 6.3 回调接口（框架上报）

### POST `/api/v1/execution/callbacks/progress`

用途：接收外部框架执行进度与结果。

Headers（建议）：
- `X-Framework-Id`: 框架标识
- `X-Event-Id`: 事件唯一 ID（幂等）
- `X-Signature`: HMAC 签名
- `X-Timestamp`: 秒级时间戳

请求体（示例）：

```json
{
  "task_id": "ET-2026-000001",
  "external_task_id": "FW-abc-123",
  "event_type": "CASE_STATUS",
  "seq": 12,
  "event_time": "2026-03-03T10:15:00Z",
  "overall_status": "RUNNING",
  "case": {
    "case_id": "TC-2026-001",
    "status": "RUNNING",
    "progress_percent": 60,
    "step_total": 10,
    "step_passed": 6,
    "step_failed": 0,
    "step_skipped": 0
  }
}
```

响应体（示例）：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "accepted": true,
    "deduplicated": false
  }
}
```

语义：
- `accepted=true, deduplicated=true`：重复事件，已幂等忽略。
- `accepted=false`：签名错误、时间窗非法、任务不存在等。

## 7. 与外部框架交互协议

## 7.1 下发到框架（平台 -> 框架）

建议字段：
- `platform_task_id`
- `callback_url`
- `cases[]`（每个包含 `case_id` 与执行所需快照）
- `runtime_config`
- `trace`（request_id / trigger_user）

## 7.2 框架回调（框架 -> 平台）

最小必填：
- `task_id`
- `event_id`
- `event_type`
- `seq`
- `event_time`

按事件类型可选：
- `overall_status`
- `case`（`case_id/status/progress/step统计`）
- `step`（若需要精细步骤落库，后续扩展）

---

## 8. 幂等、乱序与并发控制

## 8.1 幂等

- 通过 `execution_events (task_id, event_id)` 唯一索引保证。
- 重复回调直接返回 `deduplicated=true`。

## 8.2 乱序

- `ExecutionTaskCase.last_seq` 记录已处理最大序号。
- 新事件 `seq <= last_seq`：仅审计，不更新业务状态。

## 8.3 并发

- 对单 `task_id + case_id` 更新采用“条件更新”或事务：
  - 仅当 `incoming.seq > last_seq` 才更新。

---

## 9. 安全设计

## 9.1 回调验签

推荐 HMAC-SHA256：

- 签名串：`{timestamp}\n{event_id}\n{raw_body}`
- 服务端按 `framework_id -> secret` 验签
- 时间窗校验：`abs(now - timestamp) <= 300s`

## 9.2 鉴权与权限

平台用户接口（任务下发、查询）使用现有 JWT + RBAC。

新增权限建议：
- `execution_tasks:read`
- `execution_tasks:write`

回调接口不走用户 JWT，走机器验签。

---

## 10. 错误语义

- `400`：参数错误
- `401`：回调签名失败/鉴权失败
- `404`：任务不存在、用例不存在
- `409`：状态冲突（如非法状态回退）
- `422`：回调字段不完整/语义不合法
- `502`：下发外部框架失败

所有响应继续使用现有 `APIResponse` envelope。

## 13. 待确认项（实现前必须定稿）

1. 框架回调签名协议是否采用 HMAC（若已有企业网关签名方案，可替换）？
2. 回调事件是否保证全局唯一 `event_id`？若否，是否允许以 `(task_id, seq)` 做幂等主键？
3. 任务下发失败时，接口是否返回 502（并保留任务记录）？
4. 是否要求“步骤级结果”本期落库，还是仅保存步骤统计？
5. 是否需要把 execution 状态同步回 `workflow`（当前建议不同步，保持解耦）？

---

## 14. 方案结论

该方案在当前项目中以新增 `execution` 域的方式实现“任务下发 + 进度回传”，
具备以下特性：

- 与现有 `test_specs/workflow` 解耦，落地成本低。
- 回调链路具备幂等、验签、乱序保护。
- 数据结构可支撑后续报表、重试和多框架扩展。

评审通过后，可按 Phase 1 开始实现。
