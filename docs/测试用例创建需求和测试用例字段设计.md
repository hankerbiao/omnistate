
# 现代化服务器硬件测试管理平台 - 需求与用例管理模块设计方案

## 1. 架构设计理念


1. **原子化与溯源 (Traceability)**：测试需求作为独立原子，向下驱动测试用例，建立完整的需求到执行的溯源矩阵。
2. **硬件 BOM 感知 (BOM-Driven)**：需求与物理部件库（BOM/Part Number）强关联，支撑精准的硬件覆盖度（BOM Coverage）统计。
3. **动静分离与自动化解耦 (Decoupling)**：测试用例（TC）仅作为“静态设计图纸”，测试结果与统计数据归档至独立的执行表（Test Run）。自动化脚本作为独立实体与手动用例灵活绑定。

---

## 2. 核心数据模型设计 (Data Models)

### 2.1 测试需求表 (`Test_Requirement`)

**定位**：业务测试的源头，定义“要测什么硬件/功能”，由 TPM 统筹。

| 字段名称 | 数据类型 | 核心说明 |
| --- | --- | --- |
| `req_id` | String(PK) | 唯一业务编号（如：TR-2024-001） |
| `title` | String | 需求简述 |
| `description` | Text | 详细技术规范与验证目标 |
| `priority` | Enum | 优先级：P0 (阻断) / P1 / P2 / P3 |
| `platform_family` | String | 适用平台（如：Intel Eagle Stream） |
| **`target_components`** | **JSON** | **BOM 关联核心**：记录涉及的部件类型或具体的 Part Number 列表（如：`["Mem_Samsung_32G", "NVMe_PM1733"]`） |
| `tpm_owner_id` | String | TPM 负责人 ID |
| `manual_dev_id` | String | 指派的测试用例开发工程师 ID |
| `auto_dev_id` | String | 指派的自动化脚本开发工程师 ID（可选） |
| `status` | Enum | 生命周期状态：待指派 / 用例开发中 / 评审中 / 已闭环 |

### 2.2 测试用例表 (`Test_Case`)

**定位**：静态的测试逻辑图纸。**（注：不包含任何动态执行结果，测试步骤采用 JSON 存储以保持轻量化）**

| 字段名称 | 数据类型 | 核心说明 |
| --- | --- | --- |
| `case_id` | String(PK) | 唯一业务编号（如：TC-BMC-001） |
| **`ref_req_id`** | **String(FK)** | **外键**：关联的 `Test_Requirement.req_id` |
| `title` | String | 用例名称 |
| `version` | Integer | 用例版本号（如：1, 2, 3... 配合审计日志） |
| `pre_condition` | Text | **硬件 Setup**：强依赖的硬件拓扑或固件版本前置要求 |
| **`steps`** | **JSON** | **测试执行逻辑**：包含 `[{"step": 1, "action": "...", "expected": "..."}]` |
| `is_need_auto` | Boolean | TPM 评审后标识：是否需要开发自动化脚本 |
| **`script_entity_id`** | **String(FK)** | **外键**：关联的 `Auto_Script.script_id`（若已实现自动化） |
| `status` | Enum | 状态：草稿 / 待评审 / 已发布 / 废弃 |

### 2.3 自动化脚本表 (`Auto_Script`)

**定位**：独立的物理脚本库，与静态测试逻辑（TC）建立映射关系。

| 字段名称 | 数据类型 | 核心说明 |
| --- | --- | --- |
| `script_id` | String(PK) | 脚本唯一编号 |
| `repo_url` | String | 存放代码的 Git 仓库地址 |
| `file_path` | String | 仓库内的具体脚本路径（如 `/tests/memory/stress.py`） |
| `entry_point` | String | 执行入口函数或命令 |
| `env_params` | JSON | 运行时所需的环境变量配置模板 |

### 2.4 测试执行记录表 (`Test_Run`)  *(优化新增)*

**定位**：承载所有动态执行数据，保障 TC 表的纯净，支持后续高并发的数据写入。

| 字段名称 | 数据类型 | 核心说明 |
| --- | --- | --- |
| `run_id` | String(PK) | 单次执行记录的唯一 ID |
| **`ref_case_id`** | **String(FK)** | 关联的 `Test_Case.case_id` |
| `exec_type` | Enum | 执行方式：Manual (手动) / Auto (自动) |
| `dut_id` | String | 实际运行该测试的服务器/设备台账 ID |
| `hardware_config` | JSON | 运行时该设备的实际 BOM 状态快照 |
| **`result`** | **Enum** | 执行结果：Pass / Fail / Block / Skip |
| `failure_reason` | String | 失败归类标签（如：Firmware Bug, Hardware Damage） |
| `duration_seconds` | Integer | 实际执行耗时（用于效率统计） |
| `operator_id` | String | 执行人（自动化时为 System/Runner ID） |

## 3. 关键业务机制与统计实现

### 3.1 TPM 自动化流转机制 (Workflow)

1. 用例开发工程师在 TC 表中完成 `steps` 编写，将 `status` 变更为 **“待评审”**。
2. TPM 进行评审，若决定需要自动化，则将 `is_need_auto` 置为 `True`。
3. 系统捕获到该事件，自动触发一条指派给 `Test_Requirement.auto_dev_id` 的系统通知/任务。
4. 自动化工程师开发完毕后，将脚本信息录入 `Auto_Script` 表，并将生成的 `script_id` 回填至关联用例的 `script_entity_id` 字段中。

### 3.2 覆盖度统计算法模型 (Coverage Analytics)

依托动静分离的架构，系统支持以下多维统计报表：

* **静态需求覆盖度**：`Count(有对应 TC 且状态为已发布的 TR) / Count(总 TR)`
* **动态 BOM 覆盖度**：提取 `Test_Requirement.target_components` 中的目标 PN 清单，比对 `Test_Run` 表中 `hardware_config` 存在 `Pass` 记录的交集。
* *公式：针对特定硬件型号通过的测试数 / 该型号理论需要执行的总测试数。*



### 3.3 审计日志记录 (Audit Log)

系统底层拦截 TR 和 TC 表的所有 `Update` 操作，记录至公共的 `System_Audit_Log` 表中。

* **记录内容**：操作人、操作时间、目标实体 ID、变更字段（采用 `{"field": "pre_condition", "old": "A", "new": "B"}` 格式）。
* **作用**：满足硬件测试过程中的合规性审查（如 ISO/IEC 17025 等实验室标准）。
