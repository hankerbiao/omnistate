# 文档治理与重构方案

更新时间：2026-03-17

## 1. 目标

- 将 `docs/` 收敛为可长期维护的事实源，而不是临时说明的堆叠区。
- 明确每类文档的边界，减少“同一主题多份解释并行”。
- 将“当前实现”与“未来方案”分离，避免文档误导前后端联调。
- 建立后续清理和重构的执行基线。

## 2. 当前诊断

### 2.1 已有优点

- 根目录已经开始收口，存在统一入口和目录原则。
- 后端设计文档与 API 文档已经初步分层。
- 若干主题已具备“唯一维护入口”的意识，例如权限矩阵、字段规范。

### 2.2 核心问题

- 多份文档对同一主题重复描述，但内容不一致。
- 部分 API 文档与当前代码实现已经漂移。
- `docs/` 中混入构建产物、缓存、依赖目录，不符合 docs-as-code 治理要求。
- “设计方案”“当前实现”“接入说明”三类文档边界仍不够清晰。

## 3. 文档分层规则

### 3.1 根目录

根目录只保留项目级、跨模块、跨端的权威文档：

- 架构总览
- 接口接入约定
- 领域对象与字段规范
- 跨模块业务方案
- 文档治理规则

### 3.2 `api/`

只保留“当前可调用接口”的事实文档：

- 路径
- 权限
- 请求/响应
- 状态码
- 查询参数

禁止在 `api/` 中写大量未实现能力、路线图或未来方案。

### 3.3 `backend/`

只保留后端实现设计与约束：

- 鉴权与授权设计
- 权限模式
- 数据关系约束
- 落地实现说明

### 3.4 `changes/`

只做归档，不作为当前事实源。

### 3.5 新增建议目录

后续建议新增：

- `architecture/`：系统总览、运行时、部署、可观测性
- `adr/`：关键架构决策记录
- `domain/`：Requirement、Test Case、Execution、Workflow 等领域规则

## 5. 文档处置清单

### 5.1 保留并持续维护

- `README.md`
- `项目架构规范.md`
- `接口与认证说明.md`
- `测试对象与字段规范.md`
- `测试设计与BOM关联方案.md`
- `backend/permission_validation_patterns.md`
- `backend/requirement_testcase_relation_design.md`
- `changes/2026-03-02_03_阶段变更汇总.md`

### 5.2 保留但需要重写或校正

- `api/index.md`
  - 当前统一响应格式与真实实现不一致。
  - 应只保留基础约定和模块索引，不再伪造统一错误结构。

- `api/health.md`
  - 当前内容描述了未实现接口与字段。
  - 应收敛到 `/health`、`/health/ready`、`/health/live`。

- `api/execution.md`
  - 当前对象模型、状态和接口集合与实现偏差较大。
  - 应以 `backend/app/modules/execution` 当前实现为准重写。

- `backend/authorization_design.md`
  - 需补 execution 权限矩阵、navigation 权限、管理员专用接口说明。
  - 需消除与当前代码的偏差。

- `backend/rbac_design.md`
  - 应突出模型与职责，不再重复整份权限矩阵。

- `测试执行集成方案.md`
  - 仍混有“未来落地顺序”口径。
  - 应改成“设计目标 + 当前实现 + 后续演进”三段式。

### 5.3 合并或废弃候选

- `api/auth.md`
  - 倾向收缩为“认证接入说明”，避免再重复 RBAC 细节。

- `api/auth-modules.md`
  - 倾向保留为“认证授权管理 API”。
  - 与 `api/auth.md` 明确拆分：前者写接入，后者写管理接口。

如果后续仍出现大量重复，优先保留一份并废弃另一份。

### 5.4 应从版本库中清理的非源码内容

以下内容不应作为长期跟踪对象存在于仓库中：

- `docs/node_modules/`
- `docs/.vitepress/cache/`
- `docs/.vitepress/dist/`
- `docs/.DS_Store`

说明：

- 本轮先不直接删除，以避免误伤当前工作区。
- 后续应通过 `.gitignore` 和一次性仓库清理完成收敛。

## 6. 权威事实源规则

### 6.1 API 事实源

优先级应为：

1. 代码中的 `routes.py`、`schemas.py`
2. `docs/api/*.md`
3. 根目录接入说明

根目录文档不应覆盖模块级 API 细节。

### 6.2 权限事实源

优先级应为：

1. 路由依赖声明
2. `backend/authorization_design.md`
3. `api/auth-modules.md`

权限矩阵只应保留一个完整维护入口。

### 6.3 领域字段事实源

优先级应为：

1. 后端 schema / model
2. `测试对象与字段规范.md`
3. 前端表单说明

## 7. 文档模板约束

后续新增设计文档建议统一包含：

- 文档目标
- 文档边界
- 当前实现
- 约束与不变量
- 风险与已知限制
- 演进方向
- 事实源引用

后续新增 API 文档建议统一包含：

- 基础路径
- 权限要求
- 请求参数
- 响应结构
- 错误语义
- 当前实现备注

## 8. 执行顺序

### 第一阶段：纠偏

- 修正站点导航和编辑链接
- 修正总览页中的统一响应格式
- 修正健康检查文档
- 修正执行域文档
- 修正权限矩阵

### 第二阶段：收口

- 处理 `auth.md` 与 `auth-modules.md` 的边界
- 为架构与领域增加独立目录
- 将“未来方案”从“当前实现文档”中剥离

### 第三阶段：工程化

- 清理 `docs/node_modules`、`.vitepress/cache`、`.vitepress/dist`
- 补齐 `.gitignore`
- 视情况引入 OpenAPI 生成或校验流程

## 9. 本轮执行范围

本轮优先处理以下事项：

- 新增本治理文档
- 修正文档入口与站点配置中的明显错误
- 修正最关键的事实冲突页面

暂不在本轮执行：

- 大规模目录迁移
- 自动生成式 API 文档改造
- 删除已跟踪的大体积目录
