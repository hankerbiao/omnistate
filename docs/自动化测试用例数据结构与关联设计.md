# 自动化测试用例数据结构与关联设计

## 1. 目标
- 建立独立的“自动化测试用例库”（可版本化、可复用、可审计）。
- 让业务测试用例（`test_cases`）可以稳定关联自动化用例，并支持任务流指派与交付闭环。

## 2. 建议的数据模型

### 2.1 自动化测试用例主表（automation_test_cases）
建议新增独立集合：`automation_test_cases`

```json
{
  "auto_case_id": "AUTO-CASE-10023",
  "name": "BMC 冷启动后传感器数据一致性",
  "version": "1.3.2",
  "status": "ACTIVE",
  "framework": "pytest",
  "automation_type": "hardware",
  "repo_url": "git@repo:test/automation.git",
  "repo_branch": "main",
  "script_entity_id": "script_power_cycle_v3",
  "entry_command": "pytest tests/power_cycle/test_smoke.py -m smoke",
  "runtime_env": {
    "os": "ubuntu22.04",
    "python": "3.11",
    "dependencies": ["ipmitool", "redfish-client"]
  },
  "tags": ["smoke", "nightly", "bmc"],
  "maintainer_id": "u-1001",
  "reviewer_id": "u-1002",
  "description": "冷启动后关键传感器读数一致性校验",
  "assertions": [
    "温度阈值不超限",
    "传感器缺失数为 0"
  ],
  "created_at": "2026-03-02T10:00:00Z",
  "updated_at": "2026-03-02T10:00:00Z",
  "is_deleted": false
}
```

建议索引：
- 唯一：`auto_case_id + version`
- 普通：`script_entity_id`、`status`、`maintainer_id`、`tags`

### 2.2 业务测试用例表（test_cases）
当前已有字段可继续保留：
- `is_need_auto`
- `is_automated`
- `automation_type`
- `script_entity_id`
- `workflow_item_id`

建议新增明确关联字段：
- `automation_case_ref`: `{ auto_case_id: string, version?: string }`

示例：
```json
{
  "case_id": "TC-20260302-001",
  "title": "BMC 冷启动校验",
  "is_need_auto": true,
  "is_automated": true,
  "automation_type": "hardware",
  "script_entity_id": "script_power_cycle_v3",
  "automation_case_ref": {
    "auto_case_id": "AUTO-CASE-10023",
    "version": "1.3.2"
  }
}
```

## 3. 关联方式

### 3.1 推荐：弱耦合引用（字段引用）
- 在 `test_cases` 中保存 `automation_case_ref.auto_case_id/version`。
- 读取详情时由后端 join（或二次查询）返回自动化用例摘要。

优点：
- 改造成本低，兼容现有 `script_entity_id`。
- 支持 1:N（同一个自动化用例服务多个业务用例）。

### 3.2 可选：映射表（case_automation_links）
若后续需要多对多和关联历史，建议新增映射集合：
```json
{
  "case_id": "TC-20260302-001",
  "auto_case_id": "AUTO-CASE-10023",
  "auto_case_version": "1.3.2",
  "relation_type": "PRIMARY",
  "linked_by": "u-1003",
  "linked_at": "2026-03-02T10:30:00Z",
  "is_deleted": false
}
```

## 4. 任务流闭环建议
- 创建业务测试用例时，沿用当前后端逻辑生成 `workflow_item_id`。
- 指派开发人：`owner_id` 与 workflow 当前处理人保持一致。
- 开发完成后：更新 `test_cases` 的自动化字段，并记录在 workflow 日志中。
- 状态建议：`DRAFT -> ASSIGNED -> DEVELOPING -> DONE`（按现有 workflow 配置落地）。

## 5. API 建议
- `POST /api/v1/automation-test-cases`
- `GET /api/v1/automation-test-cases/{auto_case_id}`
- `GET /api/v1/automation-test-cases?framework=&status=&tag=`
- `PUT /api/v1/automation-test-cases/{auto_case_id}`
- `POST /api/v1/test-cases/{case_id}/automation-link`
- `DELETE /api/v1/test-cases/{case_id}/automation-link`

## 6. 迁移策略
1. 先加 `automation_case_ref`（可选字段），不影响旧数据。
2. 旧数据按 `script_entity_id` 回填自动化用例库。
3. 前端优先读 `automation_case_ref`，无值时回退到 `script_entity_id`。
4. 稳定后将 `script_entity_id` 定位为执行层字段，关联主键使用 `automation_case_ref`。
