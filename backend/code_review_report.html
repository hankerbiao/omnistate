📌 核心结论

架构层面良好：采用模块化分层（api/service/repository/domain/schemas），共享层（shared）抽象合理，日志系统、异常处理分层、Kafka 死信机制、路由自动注册、配置热加载等工程质量较高。

安全是最大短板：生产密钥明文提交到 Git；多个模块（execution_plan、project、ai_analysis、system_config）路由缺失鉴权；Redis 管理 API 任意登录用户可操作任意 key；附件无文件类型校验。

异步实践有硬伤：AI 客户端与 RabbitMQ 生产者在 async 上下文中调用同步阻塞 API，会卡死整个事件循环，高并发下服务不可用。

性能隐患普遍：N+1 查询、全量加载到内存、$regex 全表扫描、无流式 AI 响应等问题在大数据量下会暴露。

2. AI 客户端同步阻塞事件循环

app/shared/ai/client.py:13,85,150

AIClient 使用同步 OpenAI 客户端，在 async def chat_completion 中直接调用 client.chat.completions.create()。LLM 调用耗时数秒至数十秒，期间整个事件循环被阻塞，所有其他请求被饿死。影响 ai_analysis、failure_analysis、system_config/ai_routes 全部 AI 端点。

# client.py:13
from openai import OpenAI   # 同步客户端，非 AsyncOpenAI

async def chat_completion(...):
    response = client.chat.completions.create(...)  # 同步阻塞！
修复：改用 openai.AsyncOpenAI，或至少用 await asyncio.to_thread(client.chat.completions.create, ...) 包装。EmbeddingService 已正确用 httpx.AsyncClient，可参照。

3. RabbitMQ 生产者同步阻塞事件循环 + 线程不安全

app/shared/rabbitmq/producer.py:173-254

RabbitMQProducerManager.send_task 是同步方法，内部直接调用 pika.BlockingConnection 同步 API。从 async 路由调用会阻塞事件循环。且 _ensure_connection_ready 在连接断开时 stop()+start() 重建，多任务并发调用存在竞争条件。

修复：① 对外提供 async 接口，内部用 asyncio.to_thread 包装；② 加 asyncio.Lock 保护重连逻辑；③ 参考 Kafka 生产者的 asyncio.to_thread 封装模式。
P0

4. Redis 管理 API 越权：任意登录用户可操作任意 key

app/shared/api/routes/redis.py:43-68

/redis/set、/redis/del 仅校验登录（get_current_user），不校验任何权限。domain/entity/key_id 均由用户传入，可操作 dmlv4: 命名空间下任意 key，包括覆盖服务注册信息、缓存数据。

修复：① 增加 require_permission("system:redis:write")；② 对 key 做白名单/前缀校验；③ 生产环境建议直接下线该管理 API。

💡 整改建议

1. 安全优先：密钥泄露与鉴权缺失是两类最危险的问题，建议成立安全整改专项，先扫一遍所有路由的鉴权依赖（可用脚本批量检测缺失 Depends 的写操作端点）。

2. 异步规范：建立"async 函数内禁止直接调用同步阻塞 IO"的代码规范，CI 中用 flake8-async 或 lint 规则卡控。

3. 分层推广：以 test_case_collection 模块为模板，统一各模块分层与依赖注入风格；failure_analysis 的 AI 逻辑应从路由下沉到 service。

4. 测试补强：重点补鉴权测试（每个端点验证未认证/无权限/正常三种情况）、文件上传安全测试、AI 客户端 mock 测试。
