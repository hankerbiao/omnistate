# 测试需求与测试用例关联关系设计（方案 A：单向父子关系）

## 1. 目标与原则
- **显式溯源**：需求驱动用例，保证需求→用例的单向追溯。
- **简单可落地**：快速实现，减少复杂度，利于早期迭代。
- **查询高效**：需求下的用例查询为主场景。

## 2. 关系定义
- **一对多**：一个测试需求（Test_Requirement）可关联多个测试用例（Test_Case）。
- **单向父子**：测试用例只归属一个需求。
- **关联字段**：在 `Test_Case` 中新增 `ref_req_id` 字段。

```
Test_Requirement (req_id) 1 ---- n Test_Case (ref_req_id)
```

## 3. 数据模型字段
### 3.1 Test_Requirement
- `req_id`：需求唯一编号（业务主键）
- 其他字段保持原有设计

### 3.2 Test_Case
新增/约束字段：
- `ref_req_id: String`（必填）
  - 指向 `Test_Requirement.req_id`
  - 用例归属的“父需求”

## 4. 业务规则
1. **创建测试用例时必须校验** `ref_req_id` 是否存在。
2. **更新测试用例时如果修改 `ref_req_id`**，同样必须校验需求存在。
3. **删除需求时建议拦截**（存在关联用例则不允许删除），或提供强制级联删除机制（视业务策略）。

## 5. 查询与统计
### 5.1 需求下用例列表
```
GET /api/v1/test-cases?ref_req_id=TR-2026-001
```

### 5.2 用例对应需求
- 在用例详情中直接返回 `ref_req_id`
- 或通过 `GET /api/v1/requirements/{req_id}` 查询

### 5.3 覆盖率统计
- 统计需求下的用例数量：
  - 以 `ref_req_id` 维度聚合即可

## 6. 适用范围
- 需求驱动型的测试管理流程
- 早期阶段或单需求单归属的用例管理

## 7. 局限与演进建议
**局限**
- 不支持“一个用例覆盖多个需求”

**演进建议**
- 如需多对多，可新增关联表 `Requirement_Case_Map`
- `ref_req_id` 可作为“主需求”字段保留，保证兼容
