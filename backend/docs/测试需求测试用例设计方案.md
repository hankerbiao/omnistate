# 现代化服务器硬件测试管理平台 (TCM) - 核心架构设计方案

## 1. 架构设计理念

本平台专为服务器整机与硬件（OEM/ODM）研发测试场景打造，致力于解决传统测试管理中“软硬脱节”、“覆盖度难以量化”的痛点。核心设计理念如下：

1. **显式溯源 (Explicit Traceability)**：建立 `需求 -> 用例 -> 计划 -> 执行记录` 的单向数据流，所有关联由业务角色显式确认，保障 ISO 等合规审计要求。
2. **动静分离 (Execution Decoupling)**：测试用例（TC）仅作为“静态图纸”；动态的执行结果、耗时、报错信息全部下沉至独立的执行记录集合（Test Run），支持高并发写入与多版本追溯。
3. **物理实绩统计 (Physical Grounding)**：摒弃基于“计划”的假想统计，引入**设备资产库 (DUT)** 与 **执行期 BOM 快照**。覆盖率和通过率严格基于“此时此刻插在这台机器上的真实物料”进行计算。

### 2.4 索引与查询优化建议 (Indexing)

**原则**：对高频查询字段建立索引，避免大规模执行记录导致性能下降。

**建议索引（示例）**：
- `Test_Plan`: `plan_id`(唯一), `dut_asset_id`, `status`, `owner_id`, `created_at`
- `Test_Run_Result`: `run_id`(唯一), `plan_id`, `case_id`, `executor_id`, `runner_id`, `created_at`, `result`
- `Test_Run_Step_Result`: `step_run_id`(唯一), `run_id`, `case_id`, `step_id`, `created_at`
- `Test_Plan_Component`: `plan_id`, `part_number`

## 3. 业务流转全景 (Business Workflow)

1. **需求下发**：TPM 创建 `Test_Requirement`，设定部件覆盖目标，指派开发人员。
2. **用例编写与审核**：测试开发编写 `Test_Case`；TPM 审核并按需勾选 `is_need_auto` 触发自动化开发任务。
3. **计划排期**：TPM 创建 `Test_Plan`，输入实验室具体的服务器 `asset_id`，并**手动勾选**适用的 `case_id`。
4. **测试执行与快照**：
* 执行时，系统/测试员记录结果至 `Test_Run_Result`。
* **关键动作**：系统同时抓取该 DUT 当前的 `current_bom`，作为 `bom_snapshot` 永久封存，防止后续硬件被拔除导致统计失真。


5. **数据聚合**：基于执行记录生成各类统计看板。

## 4. 关键统计指标实现路径 (Analytics & Metrics)

得益于底层“动静分离”和“BOM 快照”的设计，平台可实时计算以下高价值指标：

### 4.1 部件执行情况与质量画像

* **逻辑**：遍历 `Test_Run_Result` 集合的 `bom_snapshot` 字段。
* **产出指标**：
* **特定物料执行次数**：某型号内存（PN: XYZ）在全量测试中被跑了多少次？
* **部件稳定性**：某型号 SSD 的综合 Pass Rate 是多少？最常在哪些用例中 Fail？



### 4.2 需求/BOM 覆盖度分析 (Coverage Rate)

* **逻辑**：`Test_Requirement` 中定义的 `target_components` (分母) vs `Test_Run_Result` 中已产生 Pass 记录的部件清单 (分子)。
* **产出指标**：
* 帮助项目经理回答：“当前阶段，BOM 清单里的物料有多少比例已经被测试计划真正验证过？”



### 4.3 用例质量洞察

* **逻辑**：按 `case_id` 聚合 `Test_Run_Result` 数据。
* **产出指标**：识别“永远 Pass”（可能无效）或“永远 Fail”（可能脚本已失效）的用例，优化测试资产。

## 5. AI 审计模块 (AI Quality Audit)

**定位**：自动评估测试用例质量，提升可执行性与覆盖率，减少重复与漏测风险。

**核心能力**：
1. **完整性检查**：是否包含必要字段（前置条件/步骤/预期结果/目标部件等）。
2. **重复检测**：基于语义与结构相似度识别重复用例。
3. **需求匹配度评估**：用例与需求目标组件/验证目标的匹配程度评分。
4. **可执行性检查**：步骤是否可操作、是否缺少环境或依赖说明。
5. **Gap Analysis**：基于需求覆盖范围与现有用例集，识别覆盖缺口。

**输出结果（示例字段）**：
- `confidence`：置信度分数（0~1）
- `risk_flags`：风险提示（如“步骤缺失”、“依赖未声明”）
- `suggestions`：优化建议列表
- `duplicate_candidates`：重复候选用例 ID 列表
- `gap_tags`：缺口标签（如某部件/场景未覆盖）

**自动复审规则**：
- 当 `confidence` 低于阈值（如 0.6）时`

| `audit_id` | String(PK) | 审计记录 ID |
| `case_id` | String(Ref) | 关联的用例 ID |
| `case_version` | Integer | 用例版本（与 `Test_Case.version` 对应） |
| `requirement_id` | String(Ref) | 关联的需求 ID（可选） |
| `confidence` | Float | 置信度分数（0~1） |
| `risk_flags` | JSON/List | 风险提示列表 |
| `suggestions` | JSON/List | 优化建议列表 |
| `duplicate_candidates` | JSON/List | 重复候选用例 ID 列表 |
| `gap_tags` | JSON/List | 缺口标签（未覆盖部件/场景） |
| `needs_review` | Boolean | 是否需要人工复审 |
| `model_info` | JSON | 模型信息（版本/策略/参数） |
| `created_at` | DateTime | 审计时间 |
