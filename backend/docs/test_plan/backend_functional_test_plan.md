## 1. 测试环境与基础设施

### 1.1 测试环境

| 环境 | 用途 | 配置 |
|------|------|------|
| 开发环境 (Dev) | 日常测试、调试 | 本地 MongoDB + RabbitMQ + MinIO |
| 集成测试环境 (CI) | CI Pipeline 自动化测试 | 容器化 MongoDB + 内存队列 |
| 预发布环境 (Staging) | 回归测试、性能测试 | 完整基础设施 |

### 1.2 测试工具

- **单元测试**: pytest + pytest-asyncio
- **集成测试**: pytest + httpx (TestClient)
- **API 测试**: httpx + 自定义 TestClient（见 `tests/integration/utils/client.py`）
- **Mock**: unittest.mock / pytest-mock（Mock MongoDB、Kafka、RabbitMQ）
- **覆盖率**: pytest-cov（目标: 业务逻辑覆盖率 >= 80%）
- **DB 隔离**: 每个测试用例独立数据库或独立集合前缀

### 1.3 测试数据约定

- 测试用户: `test_admin` (ADMIN), `test_tpm` (TPM), `test_reviewer` (REVIEWER), `test_dev` (MANUAL_DEV), `test_qa` (QA), `test_tester` (TESTER)
- 工作流类型: `REQUIREMENT`, `TEST_CASE`, `BUG`, `TASK`
- 测试需求: `TR-TEST-001`, `TR-TEST-002`
- 测试用例: `TC-TEST-001`, `TC-TEST-002`

## 2. 测试策略总览

### 2.1 测试分层

```
                    ┌─────────────────────┐
                    │  E2E 测试 (关键业务流) │
                    └──────────┬──────────┘
                               │
              ┌────────────────┴────────────────┐
              │      集成测试 (跨模块 API)       │
              └────────────────┬────────────────┘
                               │
              ┌────────────────┴────────────────┐
              │   模块集成测试 (单模块 API 层)    │
              └────────────────┬────────────────┘
                               │
              ┌────────────────┴────────────────┐
              │    单元测试 (Service + Domain)   │
              └─────────────────────────────────┘
```

### 2.2 测试优先级

| 优先级 | 定义 | 占比 |
|--------|------|------|
| P0 | 核心功能（认证、CRUD、状态流转） | 60% |
| P1 | 重要功能（搜索、批量操作、权限校验） | 30% |
| P2 | 边缘功能（统计、历史记录、缓存） | 10% |

### 2.3 测试类型

| 类型 | 覆盖内容 | 执行时机 |
|------|----------|----------|
| 单元测试 | Service 方法、Domain 规则、工具函数 | 每次提交 |
| 集成测试 | API 端点、中间件、异常处理 | 每次提交 |
| 端到端测试 | 跨模块业务流程 | 合并前 |
| 安全测试 | 权限绕过、越权访问、注入 | 每次迭代 |
| 回归测试 | 核心功能全量 | 发布前 |
